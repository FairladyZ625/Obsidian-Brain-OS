# Nightly Pipeline Guide / Nightly Pipeline 全景指南

> **English:** The complete picture of Brain OS's nightly automation pipeline — how stages connect, hand off data, and recover from failures.
>
> **中文：** Brain OS 夜间自动化流水线的完整图景——各阶段如何连接、交接数据以及从故障中恢复。

---

## Architecture Overview / 架构概览

Brain OS uses a **4-stage nightly pipeline** that runs at different times throughout the night. Each stage has a specific responsibility, and stages pass data to each other through a shared digest file.
Brain OS 使用**四阶段夜间流水线**，在夜间不同时间运行。每个阶段有特定职责，阶段之间通过共享 digest 文件传递数据。

```
01:00 ┌─────────────────────┐
      │ Knowledge Lint       │ ← knowledge-lint-weekly (Mon 01:00)
      │ 知识库 Lint          │
      └────────┬──────────────┘
               │ write to digest
02:00 ┌────────▼──────────────┐
      │ Article Integration    │ ← article-notes-integration-nightly (daily 02:00)
      │ 文章整合              │
      └────────┬───────────────┘
               │ write to digest
03:00 ┌────────▼───────────────┐
      │ Conversation Mining     │ ← conversation-mining-nightly (daily 03:00)
      │ 对话挖掘               │
      └────────┬────────────────┘
               │ write to digest
04:00 ┌────────▼────────────────┐
      │ Knowledge Amplifier      │ ← amplifier-nightly (daily 04:00)
      │ 知识放大器              │
      └────────┬─────────────────┘
               │ finalize digest
       ┌───────▼──────────┐
       │ Nightly Digest    │ ← shared output file
       │ 每日摘要          │   03-KNOWLEDGE/01-READING/04-DIGESTS/
       └───────────────────┘     nightly-digest-YYYY-MM-DD.md
```

## The Shared Digest / 共享 Digest

### Location / 位置

```
03-KNOWLEDGE/01-READING/04-DIGESTS/nightly-digest-YYYY-MM-DD.md
```

### Structure / 结构

```markdown
# Nightly Digest — YYYY-MM-DD

> Auto-generated by the nightly pipeline.

## 00: Metadata / 元数据
- Generated: YYYY-MM-DD HH:MM
- Pipeline status: ✅ normal / ⚠️ degraded / ❌ failed

## 01:00 Knowledge Lint / 知识库 Lint
(Filled by knowledge-lint-weekly)

## 02:00 Article Integration / 文章整合
(Filled by article-notes-integration-nightly)

## 03:00 Conversation Mining / 对话挖掘
(Filled by conversation-mining-nightly)

## 04:00 Amplifier / 放大器
(Filled by knowledge-flywheel-amplifier-nightly)

## Summary / 总结
(Filled by the last stage — human-facing summary)
```

### Rule / 规则

**Each stage ONLY writes to its own section. Never touches other sections.**
**每个阶段只写自己的部分。绝不碰其他部分。**

This is the core contract that makes parallel development possible.
这是让并行开发成为可能的核心契约。

## Stage Details / 各阶段详解

### Stage 1: Knowledge Lint (01:00 Mon) / 知识库 Lint（周一 01:00）

| Item / 项目 | Detail / 详情 |
|---|---|
| Cron prompt | `prompts/cron/knowledge-lint-weekly.md` |
| Frequency / 频率 | Weekly, Monday 01:00 |
| Input / 输入 | Entire vault content |
| Output / 输出 | Lint report + digest section `## 01:00` |
| What it does / 做什么 | Checks knowledge base health: broken links, orphaned notes, missing metadata |

**Key rule:** Always fetch target date from system command first.
**关键规则：**始终先从系统命令获取目标日期。

---

### Stage 2: Article Integration (02:00 daily) / 文章整合（每日 02:00）

| Item / 项目 | Detail / 详情 |
|---|---|
| Cron prompt | `prompts/cron/article-notes-integration-nightly.md` |
| Frequency / 频率 | Daily 02:00 |
| Input / 输入 | New/updated article notes in `03-KNOWLEDGE/02-WORKING/01-ARTICLE-NOTES/` |
| Output / 输出 | Updated metadata, graph edges, digest section `## 02:00` |
| What it does / 做什么 | Ingests new articles, extracts relations, updates cross-references and lightweight index |

**Handoff contract:** Reads Stage 1's lint findings to avoid re-processing known-bad notes.
**交接契约：**读取阶段 1 的 lint 发现，避免重新处理已知有问题的笔记。

---

### Stage 3: Conversation Mining (03:00 daily) / 对话挖掘（每日 03:00）

| Item / 项目 | Detail / 详情 |
|---|---|
| Cron prompt | `prompts/cron/conversation-knowledge-mining-nightly.md` |
| Frequency / 频率 | Daily 03:00 |
| Input / 输入 | Recent AI conversation transcripts |
| Output / 输出 | Synthesized knowledge entries, digest section `## 03:00` |
| What it does / 做什么 | Exports recent conversations, mines for reusable knowledge, resolves project context |

**Pre-flight check:** Verifies transcript availability and QMD health before starting.
**预检：**开始前验证 transcript 可用性和 QMD 健康。

---

### Stage 4: Amplifier (04:00 daily) / 放大器（每日 04:00）

| Item / 项目 | Detail / 详情 |
|---|---|
| Cron prompt | `prompts/cron/knowledge-flywheel-amplifier-nightly.md` |
| Frequency / 频率 | Daily 04:00 |
| Input / 输入 | Outputs from Stages 2 + 3 |
| Output / 输出 | Cross-source synthesis, final summary, digest finalized |
| What it does / 做什么 | Reads upstream outputs, finds cross-theme connections, amplifies weak signals, writes human-facing summary |

**Finalizer:** This stage also writes the `## Summary` section of the digest.
**收尾者：**此阶段还负责写 digest 的 `## Summary` 部分。

## Failure Handling / 故障处理

### Degraded Mode / 降级模式

When an upstream stage is missing or failed:
当上游阶段缺失或失败时：

```markdown
⚠️ [StageName] YYYY-MM-DD | DEGRADED: <upstream missing> | partial result available
```

The downstream stage should:
下游阶段应该：

1. Note which upstream data is missing
   记录哪些上游数据缺失
2. Continue with available data
   用可用数据继续
3. Clearly mark degraded sections in digest
   在 digest 中清晰标记降级部分
4. Never invent missing data
   绝不编造缺失数据

### Recovery / 恢复

If a stage fails completely:
如果某个阶段完全失败：

1. Next run will pick up where it left off (digest is append-friendly)
   下次运行会从中断处继续（digest 是追加友好的）
2. Manual re-run is always possible: trigger the cron prompt manually
   手动重跑总是可行的：手动触发 cron prompt
3. Check `.learnings/observer/` for recurring patterns
   检查 `.learnings/observer/` 是否有反复出现的模式

## Monitoring / 监控

### How to check pipeline health / 如何检查流水线健康度

1. Read latest digest: `03-KNOWLEDGE/01-READING/04-DIGESTS/nightly-digest-*.md`
   读最新 digest
2. Check observer report: `.learnings/observer/YYYY-MM-DD.md`
   检查观察者报告
3. Review git log for commit gaps
   查看 git log 中是否有提交间隔
4. Run Observer skill for automated analysis
   运行 Observer skill 进行自动分析

### Health indicators / 健康指标

| Indicator / 指标 | Healthy / 健康 | Warning / 警告 |
|---|---|---|
| Digest updated today? / 今天更新了？ | ✅ Yes | ❌ No update > 12h |
| All 4 stages present? / 4 个阶段都在？ | ✅ Complete | ⚠️ Missing stage(s) |
| Git commit after each stage? / 每阶段后都有 commit？ | ✅ Yes | ❌ Missing commit |
| Error count in gateway logs? / 网关日志错误数？ | 0-2 | > 5 or same error 3+ times |
