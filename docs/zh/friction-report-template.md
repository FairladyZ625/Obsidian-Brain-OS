# 摩擦报告模板

当你需要产出一份结构化摩擦报告时，使用这个模板。

## 报告头

```yaml
report_id: 自动生成
period: YYYY-MM-DD 至 YYYY-MM-DD
agent: {{MAIN_AGENT_NAME}}
status: draft | final
```

## 1. 摩擦信号

对每条信号记录：

| ID | 时间戳 | 来源 | 摘要 | 归属桶 | 是否重复？ |
|----|---------|------|------|--------|------------|
| F-001 | ... | 用户纠正 | 计划被写到 workspace 而非知识库 | reference 遗漏 | 是（第 3 次）|

### 规则
- 只收录已出现**至少两次**、或明显属于结构性的信号
- 一次性错误放在底部单独的“噪音”区

## 2. 归桶

把信号归入重复桶：

### 策：Reference 遗漏
- 信号：F-001, F-005, ...
- 模式：规则存在但没有被足够早地读取
- 数量：N

### 策：Prompt 歧义
- 信号：...
- 模式：执行说明允许多种解读
- 数量：N

（每个桶继续）

## 3. 发现

对每个有 2+ 信号的桶：
- 真正的根因是什么？
- 哪一层系统坏了？
- 是结构性的还是一次性的？

## 4. 建议回写

对每个结构性发现：

| 发现 | 层级 | 建议动作 | 优先级 |
|------|------|----------|--------|
| 计划落点错误 | AGENTS + Reference | 强化行动前检查 | 高 |

### 回写层级选项
- **Prompt**：重写执行说明
- **AGENTS**：新增或强化可见提示
- **Reference**：新增或改进可复用规则
- **Workflow**：调整扫描逻辑或自动化
- **Docs/Onboarding**：更新安装或使用指南
- **不改**：仅记本地日志，不系统化

## 5. 噪音（不系统化）

列出一次性问题及跳过原因：

| 问题 | 跳过原因 |
|------|----------|
| 输出中单次拼写错误 | 一次性，无模式 |

## 6. 下次复查日期

安排下次摩擦复查：YYYY-MM-DD
