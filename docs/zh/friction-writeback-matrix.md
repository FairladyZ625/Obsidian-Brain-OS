# 摩擦回写矩阵

用这个矩阵来判断重复问题应该回写到哪一层。

## 决策流

```
是否重复出现（2+ 次）？
├─ 否 → 仅记本地日志，结束
└─ 是 → 哪一层出了问题？
    ├─ 执行说明不清 → 重写 PROMPT
    ├─ 规则存在但被遗忘 → 强化 REFERENCE
    ├─ 规则应在行动前可见 → 更新 AGENTS
    ├─ 自动化抓不到真正问题 → 修复 WORKFLOW
    ├─ 新能力未被文档化 → 更新 DOCS/ONBOARDING
    └─ 私有内容泄露 → 修复 BOUNDARY 规则
```

## 矩阵表

| 如果问题是... | 回写到... | 示例动作 |
|---------------|----------|----------|
| Prompt 允许错误解读 | **Prompt** | 加上显式路径规则 |
| 规则存在但被跳过 | **Reference** | 强化 reference 并加 AGENTS 入口 |
| 规则应在行动前可见 | **AGENTS** | 加入强制检查清单 |
| 扫描 / cron 抓不到真正升级面 | **Workflow** | 升级扫描逻辑或 prompt |
| 新功能用户不可见 | **Docs/Onboarding** | 更新安装指南和档位 |
| 命名 / 目录误导 agent | **Reference** | 澄清命名规范 |
| 私有数据进入公开输出 | **Boundary** | 加 PII / 配置泄露检查 |
| 文档版本过旧 | **Docs** | 批量更新版本号 |
| 一次性人为错误 | **无** | 仅记 daily memory |

## 核心原则

> 不是每个错误都值得一条新规则。
> 但每个重复的结构性错误都值得一次回写。

## 反模式
- 全部回写（噪音）
- 完全不回写（停滞）
- 只改 prompt 但真问题是 workflow
- 明明可以强化已有文档却去建新的
