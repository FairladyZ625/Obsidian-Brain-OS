# 02 — 频道设计

> 为什么要分频道？怎么分？分了之后怎么协作？

---

## 为什么必须分频道

### 核心问题：上下文污染

如果你的 Agent 在一个频道里同时处理：
- 待办事项
- 文章沉淀
- 代码问题
- 知识检索

它的上下文会变得**极其混乱**。当它生成每日驾驶舱时，可能被昨天的代码调试对话干扰。当它处理文章时，可能被待办提醒打断。

**分频道的核心目的：让每个场景的 Agent 上下文都是干净的。**

### 实际效果

| 场景 | 不分频道 | 分频道 |
|------|---------|--------|
| 处理文章 | Agent 可能看到待办事项，分心 | Agent 只看文章相关上下文 |
| 生成驾驶舱 | Agent 被技术讨论干扰 | Agent 只看事务相关上下文 |
| 代码协作 | Agent 被文章处理打断 | Agent 只看代码相关上下文 |

## 我们的频道设计（参考）

以下是我们实际使用的频道分工，你可以根据自己的需求调整：

### 频道 1：个人事务管理

**用途**：待办、日程、承诺、决策、日常管理

**发给这里的消息类型**：
- "帮我记一下明天要开会"
- "这个项目进展怎么样了"
- "今天还有什么没做完的"
- "帮我排一下这周的优先级"

**Agent 职责**：
- 管理 `00-INBOX/todo-backlog.md`
- 生成每日驾驶舱
- 跟进承诺事项
- 提醒截止日期

**Agent 不应该做的**：
- 不处理文章
- 不写代码
- 不做深度研究

### 频道 2：文章沉淀

**用途**：文章处理、知识整合、阅读管理

**发给这里的消息类型**：
- 发一个文章链接："帮我处理这篇文章"
- "昨天处理的文章结果怎么样"
- "这个领域的知识卡帮我更新一下"

**Agent 职责**：
- 处理 `02-WORKING/01-ARTICLE-NOTES/` 中的文章
- 生成知识卡
- 更新主题页
- Nightly Pipeline Stage 1（02:00）

### 频道 3：代码协作

**用途**：编程、调试、技术讨论

**发给这里的消息类型**：
- "帮我看看这段代码"
- "这个 bug 怎么修"
- "帮我重构这个模块"

**Agent 职责**：
- 代码编写和调试
- 技术方案设计
- 代码审查

### 频道 4：知识检索

**用途**：在知识库中查找信息

**发给这里的消息类型**：
- "帮我找一下关于 XX 的知识卡"
- "上次讨论 XX 的时候说了什么"
- "这个领域有哪些相关的知识"

**Agent 职责**：
- 搜索知识库
- 回答知识相关问题
- 提供相关笔记链接

## 频道之间的协作

频道是隔离的，但信息需要流通。方法是：

### 1. 共享知识库

所有频道读写同一个 Obsidian Vault。Agent A 在文章沉淀频道写入的知识卡，Agent B 在事务管理频道也能读取。

### 2. Cron Job 跨频道调度

Nightly Pipeline 的三个阶段可以由不同 Agent 执行：

```
02:00 → 文章沉淀频道的 Agent → 处理文章
03:00 → 知识检索频道的 Agent → 挖掘对话
04:00 → 主 Agent → 综合放大
07:00 → 事务管理频道的 Agent → 生成驾驶舱
```

关键是：每个阶段都是 `sessionTarget: "isolated"`（独立会话），不会互相污染。

### 3. 人工中转

当频道 A 的结果需要通知频道 B 时：
- 你手动转发
- 或者配置 Cron Job 的 `delivery` 发送到目标频道

## 你应该怎么设计

### 最小配置（1 个频道）

如果你刚开始，1 个频道就够：

```
# 个人助理
所有事情都发给这个频道
```

缺点：上下文会混杂。但对于刚开始的用户来说，够用了。

### 推荐配置（2-3 个频道）

```
# 事务管理     ← 待办、日程、驾驶舱
# 知识与文章   ← 文章处理、知识检索
# （可选）代码  ← 如果你有编程需求
```

### 完整配置（4+ 个频道）

像我们一样，每个场景一个频道。适合重度用户。

## Discord 频道创建步骤

1. 在 Discord 服务器中创建频道
2. 在 OpenClaw 配置中将频道绑定到 Agent：

```json
{
  "channels": {
    "discord": {
      "accounts": {
        "main": {
          "channels": {
            "个人事务管理": "CHANNEL_ID_1",
            "文章沉淀": "CHANNEL_ID_2",
            "代码协作": "CHANNEL_ID_3"
          }
        }
      }
    }
  }
}
```

3. 在 Cron Job 的 `delivery` 中指定目标频道

## 频道设计的反模式

### ❌ 一个频道干所有事

上下文混乱，Agent 判断不准。

### ❌ 太多频道（> 10 个）

管理成本高，你自己都记不住该发到哪里。

### ❌ 频道之间没有共享知识库

每个频道各自为政，信息孤岛。

### ✅ 按职责分，2-4 个频道

清晰、好管理、上下文干净。
