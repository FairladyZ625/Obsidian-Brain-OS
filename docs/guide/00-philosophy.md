# 00 — 核心方法论

> 为什么 Brain OS 要这样设计？理解了这些，后面的一切都顺理成章。

---

## 三个核心问题

Brain OS 解决的是三个问题：

### 问题 1：知识进了 Obsidian 就死掉

大多数人用 Obsidian 的方式：看到好文章 → 保存链接 → 再也没看过。

问题在于：**保存不等于消化**。你需要的不是"收藏夹"，而是"消化系统"。

Brain OS 的解法：**Nightly Pipeline**。

你只需要把文章丢进 `02-WORKING/01-ARTICLE-NOTES/`，AI 会在凌晨 2 点自动：
1. 提取关键概念
2. 写入对应领域的知识卡
3. 更新相关主题页
4. 生成每日摘要

你早上起来看摘要就行。

### 问题 2：待办清单越写越长，但从来不执行

问题在于：**清单不分优先级、不分场景、不看截止日期**。

Brain OS 的解法：**每日驾驶舱**。

AI 每天早上 7 点读你的待办库，生成一个聚焦的"今天怎么打"：
- 最重要的 3 件事
- 今天必须推进但不做完的
- 等待反馈的
- 低能量时可以做的

你不需要自己排序，AI 帮你排好。

### 问题 3：和 AI 聊了很多，但聊完就忘

你每天都在和 AI 对话，但真正有价值的洞察往往淹没在聊天记录里。

Brain OS 的解法：**对话挖掘**。

凌晨 3 点，AI 会自动扫描最近 3 天的对话记录，提取：
- 有价值的决策和判断
- 可复用的模式
- 需要跟进的线索

## 设计原则

### 1. 捕获一次，处理永远

你的工作只有一步：**把东西丢进 INBOX**。

- 文章 → `02-WORKING/01-ARTICLE-NOTES/`
- 待办 → `00-INBOX/todo-backlog.md`
- 想法 → 同样进 INBOX

后面的整理、分类、连接，全部由 AI 完成。

### 2. 读写分离

知识库分为两层：

- **READING/** — 你看的。AI 处理好的、可直接阅读的内容。
- **WORKING/** — AI 的工作台。原始输入、草稿、候选。

你永远从 `READING/04-DIGESTS/`（每日摘要）开始阅读。不需要翻 WORKING 层。

### 3. 频道隔离

不同类型的工作用不同的 Discord 频道：

| 频道 | 职责 | 为什么分 |
|------|------|---------|
| 个人事务管理 | 待办、日程、承诺 | Agent 上下文干净，专注事务 |
| 文章沉淀 | 文章处理、知识整合 | Agent 只看文章，不被事务干扰 |
| 代码协作 | 编程、调试 | 技术上下文，不影响知识管理 |

**核心目的**：防止上下文污染。一个频道的 Agent 不应该看到另一个频道的聊天记录。

### 4. 模板大于自由

Brain OS 提供固定模板：
- 文章笔记模板（frontmatter 字段固定）
- 待办格式（P0-P3 + 字段标准化）
- 项目简介模板（5 行内说清一个项目）

模板的目的是让 AI 能**程序化处理**。如果你自由发挥格式，AI 就没法自动化。

### 5. 迭代大于完美

第一天你的系统不会完美。但如果你坚持：
- 每天把文章丢进去
- 每天看 AI 生成的摘要
- 每周调整一次 Prompt 和 Skill

一个月后，你的系统会比任何现成工具都好用。因为它是**你的**系统。

---

## 这套系统适合谁？

**适合**：
- 每天和 AI 对话，但感觉"聊了就忘"的人
- 知识收藏了很多，但从来不回头看的人
- 待办清单越来越长，但执行力不够的人
- 想让 AI 真正成为"私人助手"而不是"聊天机器人"的人

**不适合**：
- 只需要一个简单笔记工具的人（直接用 Obsidian 就够了）
- 不想花时间配置和迭代的人（这套系统需要投入）
- 不使用 AI Agent 的人（Nightly Pipeline 需要 OpenClaw）

---

## 和传统知识管理的区别

| 传统方式 | Brain OS 方式 |
|----------|--------------|
| 手动整理笔记 | AI 自动处理 |
| 知识是静态的 | 知识每天在增长和连接 |
| 你找知识 | AI 推送知识给你 |
| 待办靠自觉 | AI 每天提醒你 |
| 学到的东西容易忘 | AI 从你的对话中提取洞察 |
| 一个工具干所有事 | 频道分工，各司其职 |
